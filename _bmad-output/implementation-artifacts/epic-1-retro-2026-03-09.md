## Key Insights & Lessons Learned

### Insight #1: Acceptance Criteria Need Explicit Verification Steps

**Problem:** AC was too vague - "Then a working local server starts" didn't specify HOW to verify.

**Example:**

- ❌ Before: "Then a working local server starts"
- ✅ After: "Then `npm run build:server` exits 0 AND `dist/server/index.js` exists AND server starts AND responds to `GET /api/stories` with 200"

**Impact:** Premature completion claims, false confidence, rework needed.

**Action:** All future AC must include explicit verification steps (commands, outputs, file checks).

---

### Insight #2: Types Should Be Defined First in Single Source

**Problem:** Story 1.2 had "temporary" types in `server/lib/types.ts` that drifted from what Story 1.3 defined in `src/types/bmad.ts`.

**Example:**

- `SprintStatus` was missing `project` field in server version
- Caused test failures and confusion

**Impact:** Type drift, test failures, rework.

**Action:** Define all shared types in `src/types/` FIRST. No "temporary" type definitions. Server should import from shared types where possible.

---

### Insight #3: Better Upfront AC Reduces Code Review Iterations

**Problem:** Every story required 2-3 code review rounds. Story 1.2 had 11 issues found across 2 reviews.

**Analysis:** If AC had been clearer about verification, half those issues would have been caught by the dev BEFORE code review.

**Impact:** Felt inefficient, but caught critical issues before shipping.

**Action:** Investment in planning (clear AC with verification steps) pays off in implementation (fewer review rounds).

## Team Agreements

1. **AC Must Include Verification Steps** - Not just "X works" but "X works AND here's how to verify"

3. **"Done" Means Verified, Tested, Type-Checked, AND Reviewed** - No shortcuts
4. **Code Review Findings Are Learning Opportunities** - Not blame
5. **Performance Targets Verified With Measurements** - Not assumptions
